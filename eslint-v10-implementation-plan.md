# ESLint v10 Migration Guide

This document codifies every change required to upgrade pepr from ESLint v9 (9.36.0) to ESLint v10 and to migrate existing pepr modules without violating the [backward compatibility policy](../../BACKWARDS_COMPATABILITY.md).

---

## Breaking Changes

### Vector 1 — `@eslint/eslintrc` disappears from the dependency tree

Every module scaffolded by `pepr init` has an `eslint.config.mjs` containing:

```js
import { FlatCompat } from "@eslint/eslintrc";
```

| | ESLint v9 (current) | ESLint v10 |
| --- | --- | --- |
| How users resolve it | Transitive: `pepr` → `eslint@9` → `@eslint/eslintrc@3.3.1` | **Gone.** ESLint v10 dropped legacy-config support. |
| User impact | Works | `MODULE_NOT_FOUND` — `pepr format` crashes |

### Vector 2 — `globals` no longer transitively reachable

Both old and new templates import:

```js
import globals from "globals";
```

| | ESLint v9 (current) | ESLint v10 |
| --- | --- | --- |
| How users resolve it | Transitive: `eslint@9` → `@eslint/eslintrc` → `globals@14.0.0`. (pepr also has `globals@17.4.0` in devDependencies, but npm does not install devDeps of deps for end users.) | The transitive chain is broken. Unreachable. |
| User impact | Works | `MODULE_NOT_FOUND` — `pepr format` crashes (existing **and** newly scaffolded modules) |

### Vector 3 — `@eslint/js` is no longer transitive

The template imports:

```js
import js from "@eslint/js";
```

Under ESLint v9 this was resolved via `eslint@9` → `@eslint/js@9.36.0`. Under ESLint v10, `npm ls @eslint/js` returns empty — it is no longer bundled. Must be added to pepr's `peerDependencies`.

### Vector 4 — `@typescript-eslint/*` peer dependency conflict

The current pinned versions do not support ESLint v10:

| Package | Current version | Peer dep on eslint | ESLint v10 compatible? |
| --- | --- | --- | --- |
| `@typescript-eslint/eslint-plugin` | `8.45.0` | `^8.57.0 \|\| ^9.0.0` | **No** |
| `@typescript-eslint/parser` | `8.45.0` | `^8.57.0 \|\| ^9.0.0` | **No** |

`npm install` will fail with `ERESOLVE could not resolve` because `@typescript-eslint/eslint-plugin@8.45.0` declares `peerDependencies: { eslint: "^8.57.0 || ^9.0.0" }`, which does not include `^10.0.0`.

ESLint v10 support was added in **`@typescript-eslint/*@8.59.0`** (peer dep: `^8.57.0 || ^9.0.0 || ^10.0.0`). Both packages must be bumped together since the plugin requires `@typescript-eslint/parser: "^8.59.0"` as a peer.

### Node.js version concern

The new native flat config should **not** use `import.meta.dirname` — it requires Node >= 20.11.0, but pepr's engine field is `>=20.0.0`. Use the portable pattern instead:

```js
import { fileURLToPath } from "node:url";
import path from "node:path";
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
```

## Why Users Don't Have These as Direct Deps

The scaffolded `package.json` (generated by `genPkgJSON` in `src/cli/init/templates.ts:58-104`) only includes:

```json
{
  "dependencies": { "pepr": "...", "undici": "..." },
  "devDependencies": { "typescript": "..." }
}
```

No eslint-related packages. Currently users rely entirely on npm hoisting from pepr's dependency tree. With the peerDependencies approach (coordinated with [PR #3064](https://github.com/defenseunicorns/pepr/pull/3064)), npm 7+ auto-installs peer deps at the user's project root, so users still don't need to add them manually. Pepr's engine field (`>=20.0.0`) guarantees npm 9+, so auto-install is always available.

## Coordination with PR #3064

[PR #3064](https://github.com/defenseunicorns/pepr/pull/3064) moves `eslint`, `@typescript-eslint/eslint-plugin`, and `@typescript-eslint/parser` from `dependencies` to `peerDependencies`. The rationale is that these are lint tools used by `pepr format`, not runtime libraries.

This ESLint v10 migration **must be coordinated** with that PR because:

1. Both touch the same `package.json` dependency entries
2. Moving to `peerDependencies` changes how users resolve these packages — transitive deps from pepr's `dependencies` no longer apply; instead npm 7+ auto-installs peers at the project root
3. The packages that ESLint v10 drops from its transitive tree (`@eslint/js`, `globals`, `@eslint/eslintrc`) must also become `peerDependencies` so npm auto-installs them alongside eslint

The recommended approach: **land both changes together** in the same release. Phase 1 below reflects this combined approach.

## Implementation: Three Phases

### Phase 1 — Dependency Fixes (must ship with the ESLint v10 upgrade)

All eslint-related packages move out of `dependencies` and into `peerDependencies`, with versions bumped for ESLint v10 compatibility. The `globals` package also moves from `devDependencies` to `peerDependencies`.

npm 7+ (guaranteed by pepr's engine field `>=20.0.0` → npm 9+) auto-installs peer dependencies at the user's project root, so end users do not need to add any of these to their own `package.json`.

### Phase 2 — Deprecation Warning in `pepr format`

**File:** `src/cli/format/index.ts`

Add a deprecation check inside the existing `peprFormat` function. The detection block reads the user's `eslint.config.mjs` and warns if it still uses the legacy `FlatCompat` adapter. It goes **inside** the existing `try` block, immediately before the `new ESLint()` call.

**Important:** Do not duplicate the function. The existing `peprFormat` function is preserved — only two new imports are added at the top of the file and the detection block is inserted into the existing function body.

Below is the complete file after the change:

```ts
// SPDX-License-Identifier: Apache-2.0
// SPDX-FileCopyrightText: 2023-Present The Pepr Authors

import { ESLint } from "eslint";
import { formatWithPrettier } from "./format.helpers";

import { Command } from "commander";
import Log from "../../lib/telemetry/logger";

import { readFileSync, existsSync } from "fs";   // <-- NEW
import { resolve } from "path";                   // <-- NEW

export default function (program: Command): void {
  program
    .command("format")
    .description("Lint and format this Pepr module")
    .option("-v, --validate-only", "Do not modify files, only validate formatting.")
    .action(async opts => {
      const success = await peprFormat(opts.validateOnly);

      if (success) {
        Log.info("Module formatted");
      } else {
        process.exit(1);
      }
    });
}

/**
 * Perform linting and formatting on the module
 * @param validateOnly
 * @returns success
 */
export async function peprFormat(validateOnly: boolean): Promise<boolean> {
  {
    try {
      // --- NEW: Detect legacy FlatCompat-based eslint config and warn ---
      const eslintConfigPath = resolve("eslint.config.mjs");
      if (existsSync(eslintConfigPath)) {
        const configContent = readFileSync(eslintConfigPath, "utf-8");
        if (
          configContent.includes("@eslint/eslintrc") ||
          configContent.includes("FlatCompat")
        ) {
          Log.warn(
            "DEPRECATION: Your eslint.config.mjs uses the FlatCompat adapter from " +
              "@eslint/eslintrc, which will be removed in the next release. " +
              "Run 'npx pepr update' to migrate to native ESLint flat config.",
          );
        }
      }
      // --- END NEW ---

      const eslint = new ESLint();
      const results = await eslint.lintFiles(["./**/*.ts"]);

      // Track if any files failed
      let hasFailure = false;

      results.forEach(async result => {
        const errorCount = result.fatalErrorCount + result.errorCount;
        if (errorCount > 0) {
          hasFailure = true;
        }
      });

      const formatter = await eslint.loadFormatter("stylish");
      const resultText = await formatter.format(results, {} as ESLint.LintResultData);

      if (resultText) {
        Log.info(resultText);
      }

      // Write the fixes if not in validate-only mode
      if (!validateOnly) {
        await ESLint.outputFixes(results);
      }

      hasFailure = hasFailure || (await formatWithPrettier(results, validateOnly));

      return !hasFailure;
    } catch (error) {
      Log.error(error, `Error formatting module:`);
      return false;
    }
  }
}
```

### Phase 3 — Migrate `eslint.config.mjs` via `pepr update-templates`

Three sub-steps: replace the template file, update the `templates.ts` export, and wire it into `update-templates`.

#### 3a. Replace `src/templates/eslint.config.mjs` (the template itself)

**File:** `src/templates/eslint.config.mjs`

Replace the entire file. This is the native ESLint flat config — no `FlatCompat`, no `@eslint/eslintrc`:

```js
import typescriptEslint from "@typescript-eslint/eslint-plugin";
import tsParser from "@typescript-eslint/parser";
import path from "node:path";
import { fileURLToPath } from "node:url";
import js from "@eslint/js";
import globals from "globals";

const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);

export default [
  {
    ignores: ["**/node_modules", "**/dist"],
  },
  js.configs.recommended,
  {
    files: ["**/*.ts"],
    plugins: {
      "@typescript-eslint": typescriptEslint,
    },

    languageOptions: {
      parser: tsParser,
      parserOptions: {
        projectService: {
          allowDefaultProject: ["eslint.config.mjs"],
        },
        tsconfigRootDir: __dirname,
        sourceType: "module",
      },
      globals: {
        ...globals.node,
      },
    },

    rules: {
      ...typescriptEslint.configs.recommended.rules,
      "@typescript-eslint/no-floating-promises": "error",
    },
  },
];
```

**Key differences from the old FlatCompat template:**

| Old (FlatCompat) | New (native) |
| --- | --- |
| `import { FlatCompat } from "@eslint/eslintrc"` | Removed entirely |
| `const compat = new FlatCompat({...})` | Removed entirely |
| `...compat.extends("eslint:recommended", "plugin:@typescript-eslint/recommended")` | `js.configs.recommended` as a standalone array element |
| No explicit `files` pattern | Requires `files: ["**/*.ts"]` — ESLint v10 doesn't match `.ts` by default |
| No explicit TS rules spread | `...typescriptEslint.configs.recommended.rules` in the rules block |
| Uses `fileURLToPath` + `path.dirname` | Same (preserved for Node 20.0.0 compat) |

Since the `eslint` export in `templates.ts` (line 158-174) reads this file at build time via `readFileSync`, updating this file automatically updates what both `pepr init` and `pepr update-templates` will deploy. **No changes needed in `templates.ts`.**

#### 3b. Add `eslint.config.mjs` to the `update-templates` command

**File:** `src/cli/update/index.ts`

Add the `eslint` import to the existing import block (line 9-16):

```diff
 import {
   codeSettings,
+  eslint,
   helloPepr,
   prettier,
   samplesYaml,
   snippet,
   tsConfig,
 } from "../init/templates";
```

Then add the eslint migration step inside the `update-templates` action. Insert after the HelloPepr block (after line 90) and before the closing `}` of the `if (!opts.skipTemplateUpdate)` block:

```ts
          // Migrate eslint.config.mjs if it still uses the legacy FlatCompat adapter
          const eslintConfigPath = resolve(eslint.path);
          if (fs.existsSync(eslintConfigPath)) {
            const configContent = fs.readFileSync(eslintConfigPath, "utf-8");
            if (
              configContent.includes("@eslint/eslintrc") ||
              configContent.includes("FlatCompat")
            ) {
              await write(eslintConfigPath, eslint.data);
              Log.info(
                "Migrated eslint.config.mjs from FlatCompat to native ESLint flat config.",
              );
            }
          } else {
            // No eslint config at all — write the current template
            await write(eslintConfigPath, eslint.data);
          }
```

**Note on custom configs:** This only overwrites configs that contain `FlatCompat` or `@eslint/eslintrc`. Users who have already migrated manually or have custom configs without those strings are left untouched. However, users who added custom rules *on top of* the FlatCompat template will lose those customizations. The deprecation warning (Phase 2) gives them advance notice.

## Verification

After all changes are applied:

1. **Build pepr:**

   ```sh
   npm run build
   ```

2. **Test with an existing module that still has the old FlatCompat config:**

   ```sh
   npx pepr format
   ```

   Expected: succeeds AND prints the `DEPRECATION: Your eslint.config.mjs uses the FlatCompat adapter...` warning.

3. **Run the migration:**

   ```sh
   npx pepr update
   ```

   Expected: `eslint.config.mjs` is replaced with native flat config. Log line: `Migrated eslint.config.mjs from FlatCompat to native ESLint flat config.`

4. **Verify the migration worked:**

   ```sh
   npx pepr format
   ```

   Expected: succeeds with no deprecation warning.

5. **Scaffold a fresh module:**

   ```sh
   npx pepr init
   cd <new-module>
   npm install
   npx pepr format
   ```

   Expected: the new `eslint.config.mjs` is the native flat config (no FlatCompat). `pepr format` succeeds.

6. **Verify a module with a custom (non-FlatCompat) config is untouched:**

   ```sh
   npx pepr update
   ```

   Expected: eslint config is NOT overwritten if it doesn't contain `FlatCompat` or `@eslint/eslintrc`.
